---

## Структура документа

| Раздел | Содержание |
|--------|------------|
| Общее описание | Назначение Corpsite, URL входа |
| Авторизация | Логин, временный пароль, ссылка на Google Docs |

**Встроенные рисунки:** Рис. 1–9 (авторизация, задачи, редактирование, согласование).

---

## Что можно сохранить

- **Общая логика сценария «Задачи → отчёт по ссылке → согласование руководителем»** — соответствует модулю regular/adhoc tasks.
- **Описание вкладки «Задачи» как стартового экрана** для исполнителя.
- **Пошаговый цикл:** открыть → прикрепить «путь на отчёт» → отправить → смена статуса.
- **Роль руководителя:** «Все задачи» vs «Мои задачи», фильтр по статусу, «Согласовать» / «Отклонить».
- **Правила именования периодических задач** (weekly/monthly/yearly, report_code) — как справочный блок для admin/employee.
- **Скриншоты Рис. 2–6** — базовый happy-path исполнителя (при актуализации UI).

## Что устарело / не для публикации

| Элемент | Проблема |
|---------|----------|
| `http://localhost:3000/login` | Pilot URL; в hr-версии уже `https://mmc.004.kz/login` |
| Пароль `Corp2026!` в открытом тексте | **Не публиковать** в UG |
| `scripts/reset_pilot_password.py`, QM-логины | Dev/runbook, не user guide |
| Ссылка на Google Docs | Внешний черновик, не эксплуатационный источник |
| Жёсткая привязка к **ОВЭиПП** | Узкий pilot-контур; для employee UG нужен нейтральный язык |
| «Скопировать ссылку и перенести в браузер» для согласования | Вероятно устаревший UX; нужна сверка с текущим UI |
| Разделы про **заместителя и директора** | Не employee-роль → вынести в `management/` |
| Приложения 1–2 | В DOCX не заполнены |

---

## Чего не хватает

- Навигация по актуальному меню Corpsite (Directory, профиль, Telegram-привязка).
- Разделение **исполнитель / руководитель** на отдельные UG.
- Описание adhoc vs regular tasks, архива задач (упомянуто «2 рабочих дня» — нужна проверка).
- Ошибки доступа, что делать при отклонении отчёта.
- Скриншоты без pilot-данных и без внутренних URL.
