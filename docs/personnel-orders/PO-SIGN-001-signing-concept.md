# PO-SIGN-001 — Концепция подписания кадровых приказов

**Статус:** Concept only (не реализовано)  
**WP:** WP-PO-PRINT-003  
**Связь:** печатная форма (`corpsite-ui` print route), будущий контур ЭП / ознакомления

---

## 1. Назначение

Документ фиксирует **архитектурную подготовку** к электронному подписанию кадровых приказов.

В рамках текущего WP:

- **не** реализуется ЭЦП, скан подписи, immutable PDF, журнал подписей;
- **не** меняется runtime-модель подписания;
- печатная форма остаётся проекцией текущего статуса приказа.

**Решение по способу подписания пока не принято.**

---

## 2. Возможные этапы жизненного цикла (целевая картина)

```text
Черновик
    ↓
На согласовании
    ↓
На подпись
    ↓
Подписан
    ↓
Зарегистрирован
    ↓
Ознакомление сотрудников
    ↓
Архив
```

### Соответствие текущим системным статусам (MVP)

| Целевой этап | Текущий код статуса (MVP) | Печатная отметка |
|---|---|---|
| Черновик | `DRAFT` | ПРОЕКТ / ЖОБА |
| На согласовании | *(ещё нет отдельного статуса)* | — |
| На подпись | `READY_FOR_SIGNATURE` | НА ПОДПИСЬ / ҚОЛ ҚОЮҒА |
| Подписан | `SIGNED` | без watermark |
| Зарегистрирован | `REGISTERED` | без watermark |
| Ознакомление | *(отдельный процесс, не статус приказа)* | блок ознакомления на печати |
| Аннулирован | `VOIDED` | АННУЛИРОВАН / КҮШІ ЖОЙЫЛҒАН |
| Архив | *(будущий этап хранения)* | — |

Системные коды (`DRAFT`, `READY_FOR_SIGNATURE`, …) **не** показываются пользователю на печатной форме как есть.

## 3. Возможные варианты подписания

Только описание. **Не реализовывать** в текущем WP.

### 3.1. Государственная ЭЦП Республики Казахстан

- Подписание квалифицированной ЭЦП (НУЦ РК / эквивалент по законодательству).
- Требует интеграции с криптопровайдером, хранения сертификатов, проверки цепочки доверия.
- Юридически наиболее «жёсткий» вариант для официального приказа.

### 3.2. Внутренняя корпоративная подпись

- Подтверждение в системе уполномоченным пользователем (роль подписанта).
- Фиксация: кто, когда, с какого устройства/сессии.
- Может дополняться бумажным экземпляром или сканом.

### 3.3. Гибридная модель

- Электронное подтверждение в Corpsite + бумажный оригинал / скан.
- Или: корпоративная подпись для внутренних процессов + ЭЦП РК для отдельных категорий приказов.

### 3.4. Статус решения

> **Решение пока не принято.**  
> Выбор варианта — отдельный архитектурный и процессный WP после согласования с HR и ИБ.

## 4. Ознакомление сотрудников

Ознакомление — **отдельный процесс**, а не подпись приказа.

Подпись приказа = акт уполномоченного подписанта.  
Ознакомление = подтверждение, что сотрудник **видел** содержание приказа.

### Будущая модель (без реализации)

```text
Сотрудник
    ↓
открывает приказ
    ↓
подтверждает ознакомление
    ↓
фиксируются:
  - дата
  - время
  - способ (UI / бумага / иное)
  - история подтверждений
```

На текущей печатной форме блок «С приказом ознакомлен(а)» — место для **бумажного** ознакомления, не электронная фиксация.

## 6. Граница текущего print-контура

Печатная форма:

- отражает человекоориентированный статус документа;
- **не** изображает электронную или рукописную подпись, если система её не хранит;
- блок подписи остаётся местом для последующей фиксации или бумажного подписания.
