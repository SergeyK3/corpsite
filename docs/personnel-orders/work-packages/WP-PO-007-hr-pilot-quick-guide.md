# Кадровые приказы — краткая инструкция для HR (пилот)

> Controlled HR Pilot · WP-PO-007 · read-only UI + API-операции  
> Полный пакет: [WP-PO-007-controlled-hr-pilot-preparation.md](./WP-PO-007-controlled-hr-pilot-preparation.md)

---

## 1. Кто может работать

| Роль | Доступ |
|------|--------|
| HR head / HR enrollment manager | Журнал приказов, карточка сотрудника |
| Sysadmin | То же + техническая поддержка пилота |
| Обычный пользователь | **Нет доступа** к журналу приказов |

**Вход (локальная среда):** `/dev-login` → выбрать user_id кадровика с `has_personnel_admin` (не заблокированный аккаунт).

---

## 2. Где смотреть результат

| Что | Путь |
|-----|------|
| **Журнал приказов** | Кадровые процессы → **Приказы** (`/directory/personnel/orders`) |
| **История сотрудника** | Карточка сотрудника (import-card) → вкладка **«История»** |

Журнал показывает все приказы организации. Вкладка «История» — только события **конкретного** сотрудника, с привязкой к приказу (если apply выполнен).

---

## 3. P0-сценарии пилота (что проверяем)

| # | Тип | RU | Что должно произойти после apply |
|---|-----|-----|----------------------------------|
| 1 | `HIRE` | Приём | Событие «Приём» в истории; snapshot сотрудника актуален |
| 2 | `TRANSFER` | Перевод / смена должности | Событие «Перевод»; должность/отделение обновлены |
| 3 | `TERMINATION` | Увольнение | Событие «Увольнение»; сотрудник неактивен |
| 4 | `CONCURRENT_DUTY_START` | Совмещение (доп. ставка) | Событие «Изменение ставки»; суммарная ставка выросла |

**Важно:** в пилоте UI **не умеет** создавать/редактировать/регистрировать/apply/void приказы. Эти шаги выполняет технический оператор через API (см. §5).

## 4. Тестовые данные пилота

Префикс сотрудников: **`[PILOT-PO]`**  
Префикс приказов: **`PILOT-2026-*`**

После команды seed (§5) в системе появляются:

| Сценарий | Сотрудник (пример) | Приказ | Состояние |
|----------|-------------------|--------|-----------|
| Приём | `[PILOT-PO] Иванов…` | `PILOT-2026-HIRE-…` | ✅ уже apply |
| Перевод | `[PILOT-PO] Петров…` | `PILOT-2026-TR-…` | ✅ уже apply |
| Увольнение | `[PILOT-PO] Сидоров…` | `PILOT-2026-TERM-…` | ⏳ REGISTERED — apply на пилоте |
| Совмещение | `[PILOT-PO] Ким…` | `PILOT-2026-CON-…` | ⏳ DRAFT — register/apply или cancel |

**Проверка журнала:** открыть `/directory/personnel/orders`, в поиске ввести `PILOT-2026`.

**Проверка истории:** открыть карточку сотрудника с HR-import строкой (см. полный WP-PO-007 §4.3). У `[PILOT-PO]` карточка import-card может быть пустой — историю для них смотрите через техоператора (API) или после enroll.

## 6. Чеклист HR на сессии пилота (30–45 мин)

- [ ] Открыть журнал приказов, найти `PILOT-2026-*`
- [ ] Открыть приказ в drawer — проверить пункты и связанные события
- [ ] Отфильтровать по `employee_id` сотрудника «Петров»
- [ ] Открыть карточку сотрудника с import-card → вкладка **«История»**
- [ ] Убедиться: событие перевода, ссылка «Приказ №…», группировка по месяцу
- [ ] Попросить техоператора apply приказ `PILOT-2026-TERM-…` → обновить историю
- [ ] Попросить техоператора void одного тестового приказа → badge «Аннулировано», rollback назначения
- [ ] Зафиксировать замечания в gap-list (см. WP-PO-007 §7)

## 7. Чего пока нет (ожидаемо)

- Создание приказа из UI
- Регистрация / apply / void из UI
- Печать DOCX/PDF
- Типы отпусков (нет образцов, см. WP-PO-007 §6)

Это **не дефект пилота** — scope MVP WP-PO-004…005B = read API + read UI.
