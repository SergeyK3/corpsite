# PO-LIFECYCLE-002 — Политика удаления и аннулирования приказов

**Статус:** Concept only (код удаления / void policy **не** реализуется в этом WP)  
**WP:** WP-PO-PRINT-003  
**Связь:** PO-004 lifecycle, будущие WP по void / correcting orders / recovery

---

## 1. Архитектурный принцип (обязательный)

> **Удаление допускается только для черновиков, которые ещё не повлияли на кадровое состояние.**  
> После применения приказ становится частью кадровой истории.  
> Исправление выполняется через **аннулирование** или **корректирующий приказ**, а не через физическое удаление.  
> Физическое удаление рассматривается исключительно как **аварийная административная операция** с обязательным аудитом.

---

## 2. До применения (C1)

Приказ в статусе **`DRAFT`** (черновик / ПРОЕКТ) **может быть удалён**.

Условия (концептуально):

- приказ ещё не применён к кадровому состоянию;
- нет связанных `employee_events`, созданных применением этого приказа;
- удаление — операция кадрового оператора, не «тихая» очистка БД.

> Реализация UI/API удаления черновиков — **отдельный WP**. Здесь только политика.

---

## 3. После применения (C2)

**Удаление запрещено.**

Допустимые операции:

| Операция | Назначение |
|---|---|
| **Аннулировать** (`VOIDED`) | Отмена юридической силы приказа с сохранением истории |
| **Корректирующий приказ** | Новый приказ, исправляющий последствия ранее применённого |

История приказа и связанных событий **должна сохраняться**.

---

## 4. Employee Events (C3)

Записи `employee_events` (и производные кадровые состояния):

- **никогда не удаляются** обычными пользователями;
- не «откатываются» удалением приказа;
- корректируются через доменные операции (void / correcting order / reverse event — по будущей модели PE/PO).

---

## 5. Физическое удаление (C4)

Отдельная **административная** операция.

**Не** является обычной функцией системы.

Допустимые сценарии:

- восстановление после серьёзной ошибки данных;
- импорт с повреждением;
- очистка тестовых данных в не-production окружениях.

В production физическое удаление кадровых документов обычным администратором инфраструктуры **не допускается** (см. ролевую модель ниже).

---

## 6. Обязательный аудит (C5)

Любая чрезвычайная операция (физическое удаление, аварийное восстановление) должна фиксировать:

| Поле | Смысл |
|---|---|
| кто | субъект операции (user / role) |
| когда | timestamp |
| что удалил | сущность, id, тип |
| основание | классифицированная причина |
| комментарий | свободный текст |
| старые значения | snapshot / diff до удаления |

Без аудита операция не считается допустимой.

---

## 7. Предварительная ролевая модель (Part D)

**Не реализовывать** в текущем WP — только концепция.

### 7.1. Оператор отдела кадров

Может:

- создавать приказы;
- редактировать черновики;
- **удалять только черновики** (`DRAFT`).

### 7.2. Руководитель отдела кадров

Может:

- отправлять на подпись (`READY_FOR_SIGNATURE`);
- аннулировать;
- создавать корректирующие приказы.

### 7.3. Заместитель директора по административным вопросам

Имеет **те же полномочия**, что руководитель отдела кадров (в части кадровых приказов).

### 7.4. Системный администратор

- отвечает за инфраструктуру;
- **не должен** самостоятельно удалять кадровые документы;
- не принимает кадровых решений через «удалить из БД».

### 7.5. Специальная роль аварийного восстановления

Предусмотреть отдельную роль, например:

**Personnel Recovery Administrator**

(или эквивалент).

Назначение:

- исключительно аварийное восстановление / физическое удаление по политике C4;
- обязательный аудит (C5);
- не совмещать с повседневной ролью sysadmin без отдельного grant и контроля.

## 9. Связь с печатной формой

Аннулированный приказ на печати помечается watermark **АННУЛИРОВАН / КҮШІ ЖОЙЫЛҒАН**.  
Это визуальное следствие статуса `VOIDED`, а не операция удаления.
