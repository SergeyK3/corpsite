# Записка: единый орг-фильтр / орг-группировка для Corpsite

## 1. Цель

Сделать один общий механизм фильтрации и группировки данных по орг-структуре для разделов:

- Задачи
- Регулярные задачи
- Роли
- Должности
- Контакты
- Персонал
- Рабочие контакты
- Отделения

Исключение:

- **Группы отделений** — это справочник-источник орг-классификации, а не потребитель фильтра.

## 2. MVP-объем

На первом этапе внедряется только верхний уровень орг-классификации:

- клинические
- параклинические
- административно-хозяйственные

Поле `org_subgroup_id` закладывается в модель сразу, но в MVP не является обязательным для UI и фильтрации.

## 3. Базовый принцип

Каждая запись в целевых разделах должна быть приведена к `org_group_id`.

Для MVP этого достаточно.

Для задач базовое правило зафиксировано:

**Задача относится к орг-группе по owner-отделению** — effective owner unit задачи.

Источник owner unit (в порядке приоритета):

1. `regular_tasks.owner_unit_id` (для задач из шаблона);
2. поля unit на самой задаче (`owner_unit_id`, `org_unit_id`, `unit_id` в JSON/модели) — для adhoc.

**Не используется** текущий `tasks.executor_role_id` и `users.unit_id` исполнителя: роль исполнителя
меняется FSM при переходе report → approval, и org-фильтр должен оставаться стабильным.

## 4. Источник истины

Источник орг-классификации:

- таблица групп отделений
- таблица отделений

Связь должна позволять определить:

- к какой верхней группе относится отделение
- при необходимости — к какой подгруппе относится отделение

## 5. Целевая модель данных

На уровне модели закладывается единый орг-контекст:

- `org_group_id`
- `org_subgroup_id`
- `unit_id`

Для MVP обязательны только:

- `org_group_id`
- `unit_id`

## 6. Архитектурный подход

Реализуется не набор отдельных фильтров по страницам, а единый механизм из двух частей.

### Backend

Общий слой:

- `OrgScopeResolver` — определяет орг-контекст записи
- `apply_org_scope(...)` — применяет фильтр к list/query

### OrgScopeStrategy (MVP)

Каждый раздел подключает одну strategy. Для Tasks pilot:

- `TASK_OWNER_UNIT` — effective owner unit задачи (через `task_effective_owner_unit_sql`);
- `OWNER_UNIT` — `regular_tasks.owner_unit_id` (регулярные задачи, catch-up);
- `EXECUTOR_UNIT` — **не входит в MVP**; зарезервировано для будущих сценариев, если понадобится
  фильтр по фактическому исполнителю.

### Frontend

Один переиспользуемый компонент:

- `OrgScopeFilter`

Он должен уметь:

- показывать список верхних орг-групп
- передавать выбранный `org_group_id`
- сбрасывать фильтр
- позже расширяться до `org_subgroup_id` и `unit_id`

## 7. Пилотный объект

Первый пилотный раздел — **Задачи**.

Причины:

- уже есть реальный пользовательский сценарий
- эффект фильтра легко проверить визуально
- после этого тот же механизм можно переносить на регулярные задачи и справочники

## 8. MVP для Tasks

### Backend

Параметры list API:

- `org_group_id` — новый, фильтр по верхней группе owner-отделения;
- `org_unit_id` — существующий, без смены семантики: subtree по effective owner unit.

Логика `org_group_id`:

1. вычислить effective owner unit задачи (`task_effective_owner_unit_sql`);
2. join `org_units` по этому `unit_id`;
3. сравнить `org_units.group_id` с `:org_group_id`;
4. вернуть только задачи, попадающие в выбранную группу.

Логика `org_unit_id` (без изменений семантики):

- effective owner unit ∈ subtree выбранного `org_unit_id`.

Оба параметра комбинируются через **AND**.

Strategy: `TASK_OWNER_UNIT` (через `apply_org_scope`).

Явно не используется:

- `tasks.executor_role_id`;
- `users.unit_id` по текущей роли исполнителя;
- `EXECUTOR_UNIT`.

### Frontend

На странице задач добавить над списком фильтр:

- Все
- Клинические
- Параклинические
- Административно-хозяйственные

На первом этапе этого достаточно.

## 9. Перенос на остальные разделы

После Tasks тот же механизм подключается без изменения общей архитектуры:

- Регулярные задачи — по `owner_unit_id` шаблона (`OWNER_UNIT`)
- Должности — по отделению
- Персонал — по отделению сотрудника
- Контакты — по связанному орг-контексту
- Рабочие контакты — read-only витрина, но фильтруется тем же орг-контекстом
- Отделения — напрямую по своей группе
- Роли — по орг-контексту применения роли

## 10. Что не входит в MVP

На первом этапе не реализуются:

- древовидный фильтр по подгруппам
- многоуровневая группировка в левом меню
- отдельная бизнес-логика фильтрации для каждого реестра
- ручные хардкод-списки на фронте
- разные механизмы фильтрации для разных страниц
- фильтрация задач по текущему исполнителю (`tasks.executor_role_id` / `EXECUTOR_UNIT`)

## 11. Критерий готовности MVP

MVP считается готовым, если одновременно выполнены условия:

1. в разделе **Задачи** есть единый орг-фильтр по верхней группе;
2. backend фильтрует задачи по орг-группе через **effective owner unit** (`TASK_OWNER_UNIT`), а не через
   текущий `tasks.executor_role_id`;
3. механизм реализован как переиспользуемый слой, а не как локальный хак страницы задач;
4. модель допускает дальнейшее расширение до `org_subgroup_id` и `unit_id`.
