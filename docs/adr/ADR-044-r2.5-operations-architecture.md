## 1. Problem statement

R2.4 closed the **review-gated execute path**: preview → confirm → apply APPROVED links with full journal and `USER_EMPLOYEE_LINKED` audit.

Production validated:

- Execute preview and execute apply
- Audit trail on `identity_reconciliation_runs` + `user_linkage_execute_items`
- `rollback_payload` on every `LINK`/`APPLIED` item
- Idempotent re-execute (`NOOP_ALREADY_LINKED` on second run against same preview)

**Remaining operational gaps** (post R2.4f):

| Gap | Example |
|-----|---------|
| No manual override | HR must link user ↔ employee without waiting for preview match |
| No controlled unlink | Mistaken link requires L1 rollback tooling, not ad-hoc SQL |
| No item rollback API | `rollback_payload` stored but no reversible operation |
| No targeted repair | Operator cannot refresh classification for one user/employee |
| No execute history API | Runs/items queryable only via SQL |
| Incomplete concurrency story | Per-preview RUNNING guard exists; global mutex not implemented |
| Manual ops vs review path | Two write paths need unified journal + distinct permissions |

R2.5a defines the **Operations layer**: lifecycle management APIs and audit model on top of the R2.4 journal, without UI.

## 2. Design principles

| Principle | R2.5 policy |
|-----------|-------------|
| Single journal | All linkage writes flow through `identity_reconciliation_runs` + `user_linkage_execute_items` |
| Append-only history | Never DELETE journal rows; unlink/rollback add compensating rows |
| Explicit actor + reason | Every mutating op records `actor_user_id`; unlink/rollback require `reason` |
| Before/after snapshots | Every mutating item stores `before_user_snapshot` + `after_user_snapshot` |
| Fail closed | FK, `uq_users_employee_id`, inactive employee, lock conflict → fail item, not guess |
| Idempotent ops | Re-run same rollback/unlink/re-execute → NOOP, not error |
| Review path preserved | Execute path unchanged; manual link is a **separate operation**, not a bypass hidden in execute |
| No batch repair | Repair and manual ops are **single-target** only |
| Personnel admin baseline | All ops under `require_personnel_admin_api`; destructive ops optionally gated tighter |
| No schema in R2.5a | This ADR proposes DDL for R2.5b; R2.5a ships documentation only |

## 3. Current state (R2.4 baseline)

### 3.1. Run journal — `identity_reconciliation_runs`

| Column | R2.4 usage |
|--------|------------|
| `phase` | `'R2'` |
| `operation` | `USER_LINKAGE_EXECUTE_PREVIEW` \| `USER_LINKAGE_EXECUTE` |
| `dry_run` | `true` for preview; `false` for apply |
| `actor_user_id` | Operator |
| `status` | `running` → `completed` \| `failed` |
| `summary` | JSONB counts; execute runs store `source_preview_run_id` |
| `started_at` / `finished_at` | Run lifecycle |

### 3.2. Line items — `user_linkage_execute_items`

| Column | R2.4 usage |
|--------|------------|
| `run_id` | Parent run |
| `user_id` | Target user |
| `proposed_employee_id` | Intended employee |
| `source_decision_id` | Latest APPROVE at apply time (nullable for NOOP paths) |
| `action` | `LINK`, `NOOP_ALREADY_LINKED`, `SKIP_*`, `FAIL_*` (CHECK-enumerated) |
| `status` | `PLANNED` (preview), `APPLIED` \| `SKIPPED` \| `FAILED` (execute) |
| `reason_codes` | Machine codes (`PREVIEW_DRIFT`, etc.) |
| `preview_snapshot` / `decision_snapshot` | Point-in-time context |
| `before_user_snapshot` / `after_user_snapshot` | User row before/after write |
| `rollback_payload` | On `LINK`/`APPLIED` only: `{ user_id, previous_employee_id, employee_id, decision_id }` |

### 3.3. Security audit

| Event | When |
|-------|------|
| `USER_EMPLOYEE_LINKED` | Each successful execute `LINK`/`APPLIED` |

Metadata includes `user_id`, `employee_id`, `previous_employee_id`, `decision_id`, `run_id`, `match_strategy`, `classification`.

**Not yet implemented:** `USER_EMPLOYEE_UNLINKED`, manual-link metadata discriminator, rollback compensating events.

### 3.4. Implemented concurrency (R2.4d)

```text
_preview_execute_in_progress(preview_run_id):
  EXISTS identity_reconciliation_runs
  WHERE phase='R2'
    AND operation='USER_LINKAGE_EXECUTE'
    AND status='running'
    AND summary->>'source_preview_run_id' = preview_run_id
```

Per-item apply uses `SELECT … FROM users WHERE user_id = :id FOR UPDATE` inside its own transaction.

**Not yet implemented:** global advisory lock, stale-RUNNING recovery, mutex across manual ops.

## 5. Operation catalog

### 5.1. Manual Link

**Purpose:** Explicitly set `users.employee_id` for one user → one employee without requiring a prior review APPROVE.

**Use cases:** Emergency access fix, login pattern mismatch, HR-confirmed match where preview classifies `IMPOSSIBLE`, post-incident forward fix.

| Aspect | Design |
|--------|--------|
| Scope | Exactly one `user_id` + one `employee_id` per request |
| Preconditions | User active; `employee_id IS NULL` OR fail `FAILED_CONFLICT_EXISTING_LINK`; employee operational; `uq_users_employee_id` clear |
| Review gate | **Bypassed** — operator reason is the human gate |
| Optional review sync | **Recommended:** append `user_linkage_review_decisions` row with `decision=APPROVE`, `reason='[MANUAL_LINK] …'`, `proposed_employee_id` = target — preserves review audit continuity without rewriting history |
| Write | `UPDATE users SET employee_id = :employee_id WHERE user_id = :id AND employee_id IS NULL` |
| Run `operation` | `USER_LINKAGE_MANUAL_LINK` (proposed R2.5b CHECK extension) |
| Item `action` | `MANUAL_LINK` (proposed) |
| Item `status` | `APPLIED` \| `FAILED` |
| `rollback_payload` | Same shape as execute: `{ user_id, previous_employee_id, employee_id, decision_id: null }` |
| Security audit | `USER_EMPLOYEE_LINKED` with `metadata.operation = 'MANUAL_LINK'`, `metadata.reason`, `metadata.manual = true` |
| Idempotency | If already linked to same employee → `NOOP_ALREADY_LINKED`/`SKIPPED`; different employee → `FAILED` |

**Mandatory request fields:** `user_id`, `employee_id`, `reason` (min 10 chars).

**Optional:** `ticket_ref`, `confirm` boolean (UI double-submit guard in later phase).

---

### 5.2. Manual Unlink

**Purpose:** Remove `users.employee_id` linkage while preserving full history.

| Aspect | Design |
|--------|--------|
| Scope | Exactly one `user_id` per request |
| Preconditions | User exists; `employee_id IS NOT NULL` |
| `reason` | **Mandatory** (min 10 chars) — reject 400 if absent |
| Write | `UPDATE users SET employee_id = NULL WHERE user_id = :id AND employee_id = :expected` |
| Run `operation` | `USER_LINKAGE_MANUAL_UNLINK` |
| Item `action` | `MANUAL_UNLINK` |
| Item `status` | `APPLIED` \| `SKIPPED` (noop) \| `FAILED` |
| `rollback_payload` | `{ user_id, previous_employee_id: <former>, employee_id: null, source_item_id: null }` |
| Security audit | **`USER_EMPLOYEE_UNLINKED`** (new event type, R2.5b migration) with `metadata.reason`, `metadata.previous_employee_id`, `metadata.run_id` |
| History | Journal row is permanent; never DELETE prior LINK items |
| Idempotency | If already unlinked → `NOOP_ALREADY_UNLINKED`/`SKIPPED` |

**Optional review sync:** append `REJECT` decision with `reason='[MANUAL_UNLINK] …'` so execute path will not re-link without fresh APPROVE.

---

### 5.3. Rollback Execute Item (L1)

**Purpose:** Reverse a single successful execute link using stored `rollback_payload`.

| Aspect | Design |
|--------|--------|
| Input | `item_id` of source row where `action='LINK'` AND `status='APPLIED'` AND `rollback_payload` non-empty |
| Verification | Live `users.employee_id` must equal `rollback_payload.employee_id`; else `FAILED_STATE_MISMATCH` (someone manual-unlinked or re-linked) |
| Write | `UPDATE users SET employee_id = rollback_payload.previous_employee_id` (typically NULL) |
| Run `operation` | `USER_LINKAGE_ROLLBACK_ITEM` |
| Item `action` | `ROLLBACK_LINK` |
| Item `status` | `APPLIED` \| `SKIPPED` (idempotent noop) \| `FAILED` |
| `summary.source_item_id` | FK reference to rolled-back item |
| Security audit | `USER_EMPLOYEE_UNLINKED` if restoring to NULL; if restoring to prior non-null employee, `USER_EMPLOYEE_LINKED` with `metadata.compensating = true` |
| Idempotency | Track `summary.source_item_id` on prior rollback runs; second rollback of same source → `NOOP_ALREADY_ROLLED_BACK`/`SKIPPED` |
| Reversibility | Rollback of rollback (re-link) is **out of scope** — use manual link or re-execute |

**Mandatory request fields:** `reason` (min 10 chars).

**Never:** mutate or delete the source `APPLIED` item row.

---

### 5.4. Repair Linkage

**Purpose:** Re-run reconciliation **diagnosis** for a single user or employee — refresh preview classification and drift vs latest review decision.

| Aspect | Design |
|--------|--------|
| Scope | Exactly one of `user_id` **or** `employee_id` (mutually exclusive) — **no batch** |
| Writes | **None** to `users.employee_id`; optional journal row with `dry_run=true` |
| Run `operation` | `USER_LINKAGE_REPAIR` |
| Behaviour | 1) Load latest review decision for user(s) in scope; 2) `run_user_linkage_preview` filtered to target; 3) Compute `execute_ready`, `drift_reason`, blocking reason codes |
| Employee-scoped repair | Resolve linked user via `users.employee_id = :employee_id`; if none, report unlinked employee + candidate users from preview |
| Output | Repair report: fresh classification, decision state, drift flags, suggested next action (`RE_REVIEW`, `EXECUTE_READY`, `MANUAL_LINK_CANDIDATE`) |
| Idempotency | Read-only effect on linkage state; each call may create a diagnostic run row |

**Does not:** auto-execute, auto-approve, or batch-scan the cohort.

---

### 5.5. Re-run Execute

**Purpose:** After review updates (new APPROVE, drift resolved), reapply the review-gated execute path without operator manually calling preview then execute.

| Aspect | Design |
|--------|--------|
| Scope | `user_id` (single) **or** `user_ids[]` (bounded, max 50) — not full-cohort silent batch |
| Behaviour | 1) `build_user_linkage_execute_preview` for scope; 2) if `execute_allowed`, auto-execute with generated `confirm_token`; else return preview-only report |
| Run operations | Creates preview run + optional execute run (same as R2.4) |
| `summary.trigger` | `'RE_EXECUTE'` |
| Idempotency | Inherited from R2.4: second re-execute against unchanged state → `NOOP_ALREADY_LINKED` |
| Stale preview | If decision changed to REJECT/DEFER between preview and execute inside same request, apply step skips (existing R2.4d behaviour) |

**Distinction from Repair:** Repair is diagnostic only; Re-run Execute performs the review-gated write path.

**Existing endpoints remain:** operators may still call `POST …/execute-preview` + `POST …/execute` separately.

## 8. Audit model

Three layers — extends R2.4 §8 without replacing it.

### 8.1. Run journal extensions (R2.5b DDL)

Proposed new `identity_reconciliation_runs.operation` values:

| Operation | `dry_run` | Mutates `users.employee_id` |
|-----------|-----------|----------------------------|
| `USER_LINKAGE_MANUAL_LINK` | false | Yes |
| `USER_LINKAGE_MANUAL_UNLINK` | false | Yes |
| `USER_LINKAGE_ROLLBACK_ITEM` | false | Yes |
| `USER_LINKAGE_REPAIR` | true | No |
| `USER_LINKAGE_RE_EXECUTE` | false | Optional (via nested execute) |

Proposed new `user_linkage_execute_items.action` values:

| Action | Typical status |
|--------|----------------|
| `MANUAL_LINK` | `APPLIED` |
| `MANUAL_UNLINK` | `APPLIED` |
| `ROLLBACK_LINK` | `APPLIED` |
| `NOOP_ALREADY_UNLINKED` | `SKIPPED` |
| `NOOP_ALREADY_ROLLED_BACK` | `SKIPPED` |
| `FAILED_STATE_MISMATCH` | `FAILED` |

**Run `summary` conventions:**

| Key | Usage |
|-----|-------|
| `reason` | Operator reason (manual ops) |
| `ticket_ref` | Optional external reference |
| `source_item_id` | Rollback target |

## 9. Rollback model

Aligned with [R2 discovery §6.5](./ADR-044-r2-user-linkage-discovery.md) and [R2.4 §9](./ADR-044-r2-phase-r2-4-execute-engine-design.md).

| Level | R2.5 operation | Mechanism |
|-------|----------------|-----------|
| **L1 — per user / per item** | Rollback Execute Item, Manual Unlink | `rollback_payload` verification + compensating journal row |
| **L2 — per run** | Future `POST …/operations/rollback-run/{run_id}` (R2.5d+) | Reverse all `LINK`/`APPLIED` items in reverse `item_id` order |
| **L3 — snapshot** | Ops runbook only | CSV / `pg_restore` — not API |

### 9.1. Execute item rollback algorithm

```text
LOAD source_item WHERE item_id = :id
ASSERT source.action = 'LINK' AND source.status = 'APPLIED'
ASSERT source.rollback_payload.user_id = source.user_id
LOAD user FOR UPDATE
IF user.employee_id IS DISTINCT FROM rollback_payload.employee_id:
  FAIL FAILED_STATE_MISMATCH
IF user.employee_id = rollback_payload.previous_employee_id:
  NOOP (already restored)
IF EXISTS prior rollback run with summary.source_item_id = :id AND status APPLIED:
  NOOP ALREADY_ROLLED_BACK
UPDATE users SET employee_id = rollback_payload.previous_employee_id
INSERT journal item action=ROLLBACK_LINK status=APPLIED
EMIT USER_EMPLOYEE_UNLINKED or compensating USER_EMPLOYEE_LINKED
```

### 9.2. Rollback payload lifecycle

| Stage | `rollback_payload` |
|-------|-------------------|
| Execute preview | `null` / `{}` |
| Execute apply | `{ user_id, previous_employee_id, employee_id, decision_id }` |
| Manual link | Same shape; `decision_id` null unless sync_review_decision |
| After rollback | New item stores inverse payload for audit; source payload **immutable** |

## 10. Concurrency model

### 10.1. RUNNING lock semantics

| Rule | Detail |
|------|--------|
| Run creation | INSERT run with `status='running'` before any item processing |
| Finalization | UPDATE `status='completed'|'failed'`, `finished_at=now()` |
| Stale RUNNING | If process crash leaves `running` > 30 minutes, ops query flags orphan; recovery job (R2.5f) marks `failed` with `summary.error='STALE_RUNNING'` — **never auto-apply** |
| Execute per preview | At most one `USER_LINKAGE_EXECUTE` with `status='running'` per `source_preview_run_id` (implemented R2.4d) |
| Manual ops | At most one `running` mutating op per `user_id` scope (EXISTS check + advisory lock) |

### 10.2. `pg_advisory_lock` strategy

Use **transaction-scoped** locks (`pg_advisory_xact_lock`) — released on COMMIT/ROLLBACK.

| Lock key | Hash input | Purpose |
|----------|------------|---------|
| Global execute | `'adr044:r2:linkage:execute'` | One batch execute / re-execute at a time |
| Per user | `'adr044:r2:linkage:user:' \|\| user_id` | Serialize manual link/unlink/rollback/apply on same user |
| Per employee slot | `'adr044:r2:linkage:employee:' \|\| employee_id` | Serialize competing links to same employee |

**Acquisition order (deadlock avoidance):** global → user → employee (consistent ordering).

```python
# Proposed pattern (implementation sketch)
pg_advisory_xact_lock(hashtext(:lock_name))
SELECT user_id, employee_id FROM users WHERE user_id = :id FOR UPDATE
```

### 10.3. Concurrent scenario matrix

| Scenario | Result |
|----------|--------|
| Two execute workers, same preview | Second rejected: `preview run execute already in progress` (existing) |
| Execute + manual link, same user | Second blocks on user advisory lock + FOR UPDATE; re-check → noop or conflict |
| Manual unlink + rollback same user | Serialized; second op sees updated state |
| Repair + execute | Repair is read-only; no lock conflict on user row |
| Approve during manual link | Allowed; manual link does not read APPROVE gate |

### 10.4. HTTP 409 policy

Return **409 Conflict** when:

- Global execute lock unavailable after `NOWAIT` attempt (optional optimization)
- Concurrent mutating run on same user detected

Return **400** for business rule violations (not concurrency).

## 11. Safety guarantees

| # | Guarantee | Enforcement |
|---|-----------|-------------|
| O1 | No silent manual link | Reason mandatory; audit + journal |
| O2 | No silent unlink | Reason mandatory; `USER_EMPLOYEE_UNLINKED` |

| O12 | Personnel admin baseline | All routes guarded |

| 2 | Call `execute_user_linkage_from_preview(source_preview_run_id, confirm_token)` — full R2.4 validation |
| 3 | Create envelope run `operation=USER_LINKAGE_RERUN_EXECUTE`, `summary.trigger='RE_EXECUTE'` |
| 4 | Insert `RERUN_EXECUTE` items referencing nested `execute_run_id` |
| 5 | Return composite `{rerun_run_id, execute_run_id, execute: …}` |

Inherited R2.4 safety: confirm token SHA256, blocking gates, per-preview RUNNING guard, per-item fail-closed apply, `NOOP_ALREADY_LINKED` idempotency.

### 20.4. Endpoint contracts

**POST** `/admin/personnel/identity/user-linkage/operations/repair-preview`

```json

```

```json

  "source_preview_run_id": 884,
  "confirm_token": "sha256:…",
  "reason": "Re-apply after fresh APPROVE following drift fix"
}
```

Auth: `require_personnel_admin_api` (`401` / `403`).

### 20.5. Limitations

- Repair does not auto-execute or mutate linkage
- Re-run requires a **completed** R2.4 execute-preview run and valid confirm token
- No batch repair / cohort re-run
- No UI in R2.5f
