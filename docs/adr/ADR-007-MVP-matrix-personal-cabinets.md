# ADR-007 — MVP-матрица экранов личных кабинетов (v1)

## Статус
Принято

## Дата
2026-01-12

---

## Контекст

Система управления задачами и отчетностью предназначена для ежедневной операционной и управленческой работы.  
На этапе MVP необходимо обеспечить:

- минимально достаточный набор экранов;
- четкое ролевое разделение;
- быстрый доступ к ключевым действиям пользователя;
- отсутствие избыточной функциональности.

На данном этапе принято осознанное решение ограничиться двумя ролями:

- Исполнитель  
- Руководитель  

Остальные роли (Директор, Заместитель директора, Администратор) **исключены из MVP** и будут рассмотрены на последующих этапах.

---

## Цель ADR

Зафиксировать **MVP-матрицу экранов личных кабинетов** без детализации API и контрактов, чтобы:

- визуализировать будущий пользовательский интерфейс;
- синхронизировать понимание между бизнесом и разработкой;
- задать устойчивую основу для дальнейшего UX и архитектурного проектирования.

---

## Принятое решение

Для MVP v1 реализуется **4 ключевых экрана** — по 2 экрана для каждой роли.

---

## MVP-матрица экранов

### Роль: Исполнитель

| Код | Экран | Назначение | Ключевой пользовательский вопрос |
|----|------|------------|----------------------------------|
| E-1 | Список задач исполнителя | Основной экран при входе | Что мне нужно сделать? |
| E-2 | Карточка задачи | Работа с конкретной задачей | Как отчитаться по задаче? |

---

### Роль: Руководитель

| Код | Экран | Назначение | Ключевой пользовательский вопрос |
|----|------|------------|----------------------------------|
| R-1 | Задачи, требующие решения | Основной рабочий экран | Где нужно мое решение? |
| R-2 | Карточка задачи | Принятие управленческого решения | Принять или вернуть? |

---

## Детализация экранов  
(структура: Экран → Блок → Смысл)

---

### E-1. Список задач исполнителя

**Назначение:**  
Первый экран при входе в систему (типовой утренний сценарий).

**Основные блоки:**
- список всех задач исполнителя;
- визуальное выделение задач, требующих действия (например, ожидание отчета, просрочка);
- базовая сортировка по срокам.

**Ключевой смысл:**  
Исполнитель должен быстро понять:
- какие задачи у него есть;
- какие требуют немедленного внимания.

---

### E-2. Карточка задачи (исполнитель)

**Основные данные:**
- текст задачи / требования;
- текущий статус;
- срок выполнения;
- комментарии руководителя;
- история изменений и событий.

**Основное действие:**
- отправка отчета (ссылка + комментарий).

**Ключевой смысл:**  
Исполнитель понимает:
- что именно от него требуется;
- в каком формате и состоянии ожидается отчет.

---

### R-1. Задачи, требующие решения (руководитель)

**Назначение:**  
Главный рабочий экран руководителя.

**Основные блоки:**
- список задач со статусом «Ожидается согласование»;
- сортировка по срокам и приоритету;
- визуальная индикация срочности.

**Важно:**  
Экран не является списком всех задач, а отражает только те, где требуется управленческое действие.

**Ключевой смысл:**  
Руководитель сразу видит:
- где требуется его решение;
- какие задачи нельзя откладывать.

---

### R-2. Карточка задачи (руководитель)

**Основные данные:**
- заголовок и описание задачи;
- исполнитель;
- последний отчет (ссылка);
- комментарий исполнителя;
- история событий по задаче.

**Ключевые действия:**
- принять отчет;
- вернуть задачу на доработку с комментарием.

**Ключевой смысл:**  
Руководитель принимает управленческое решение и завершает цикл обработки задачи.

---

## Осознанно исключено из MVP

Следующие элементы **не входят в MVP v1** и будут рассмотрены отдельно:

- управленческие дашборды и KPI;
- аналитика и отчеты;
- массовые операции;
- расширенные фильтры;
- роли директора и заместителя;
- административный интерфейс;
- настройки и справочники.

---

## Обоснование решения

- MVP фокусируется на ежедневных сценариях, а не на редких управленческих операциях.
- Каждый экран отвечает на один главный вопрос пользователя.
- Минимизация экранов снижает:
  - сложность UX;
  - стоимость реализации;
  - архитектурные риски.

---

## Следующие шаги (вне рамок данного ADR)

- детализация экранов на уровне блоков данных;
- описание пользовательских сценариев (User Flow);
- сопоставление экранов с backend-сущностями;
- проектирование API и контрактов.

---

## Итог

MVP v1 включает:

- 2 роли;
- 4 экрана;
- 2 ключевых пользовательских сценария:
  - выполнение задачи;
  - принятие управленческого решения.

Данный ADR является базовой точкой отсчета для дальнейшего проектирования интерфейсов и архитектуры системы.
