# ADR-031 — Контракт по разделу Directory: отделения, персонал, контакты, должности и рабочие контакты

## Статус
Принято

## Дата
2026-03-09

## Контекст

В Corpsite уже существует рабочий контур справочников и оргструктуры:

- группы отделений;
- отделения;
- роли;
- список сотрудников отделения при выборе узла оргструктуры;
- RBAC-ограничения на просмотр.

На текущем этапе требуется зафиксировать целевую модель данных и интерфейсов для кадрового и справочного блока, чтобы не смешивать:

- кадровую запись сотрудника;
- контакт человека;
- системную роль;
- должность;
- производные рабочие списки контактов.

Также требуется определить целевую структуру левого меню и центральных экранов.

---

## Проблема

Сейчас в системе фактически пересекаются несколько разных сущностей:

1. человек как контакт;
2. сотрудник как кадровая запись в отделении;
3. системная роль как набор прав;
4. должность как штатная / организационная сущность;
5. рабочие контакты как объединённое представление активных сотрудников с контактными данными.

Без явного контракта появляется риск:

- смешения ролей и должностей;
- дублирования CRUD-экранов;
- появления параллельных источников истины;
- неясности, где именно должен выполняться кадровый CRUD;
- усложнения RBAC и интерфейса.

---

## Решение

### 1. Зафиксировать базовые сущности

В рамках раздела Directory используются следующие сущности.

#### 1.1. Department Groups
Группы отделений.

Назначение:
- верхний уровень классификации оргструктуры.

Примеры:
- Клинические
- Параклинические
- Адмхоз

#### 1.2. Org Units
Отделения.

Назначение:
- иерархия организационных единиц;
- привязка персонала к конкретному отделению;
- навигация по оргструктуре;
- точка входа для просмотра сотрудников отделения.

#### 1.3. Positions
Должности.

Назначение:
- штатная / организационная сущность;
- используется в кадровом учёте;
- не равна системной роли.

Примеры:
- заведующий отделением;
- врач-онколог;
- аналитик;
- эксперт;
- экономист.

#### 1.4. Roles
Системные роли.

Назначение:
- определяют права в системе;
- участвуют в RBAC, задачах, согласованиях, маршрутизации.

Примеры:
- ADMIN
- DIRECTOR
- QM_COMPLAINT_PAT

#### 1.5. Personnel
Персонал.

Назначение:
- кадровая запись сотрудника в контексте отделения;
- отражает штатное пребывание сотрудника в организации.

Минимальные поля:
- personnel_id / employee_id;
- full_name;
- tab_no;
- org_unit_id;
- position_id;
- date_from;
- date_to;
- rate;
- is_active;
- приказ о приёме / переводе / увольнении — по возможности.

#### 1.6. Contacts
Контакты.

Назначение:
- контактная сущность человека;
- независима от конкретной кадровой записи.

Минимальные поля:
- contact_id;
- full_name;
- phone;
- email — при наличии;
- telegram_username;
- telegram_numeric_id;
- is_active.

#### 1.7. Working Contacts
Рабочие контакты.

Назначение:
- производная рабочая витрина;
- объединяет активный персонал и контакты;
- используется для просмотра, поиска и связи.

Источник:
- Personnel + Contacts

Статус:
- не является первичной кадровой сущностью;
- не является первичным справочником для ввода;
- является read-моделью или сервисным представлением.

---

## Нормативное разграничение сущностей

### Должность != Роль

Это обязательное правило.

#### Должность
Оргштатная сущность:
- кем человек работает в структуре.

#### Роль
Системная сущность:
- какие права человек имеет в системе.

Один и тот же сотрудник может иметь:
- должность: заведующий отделением;
- роль: DIRECTOR или SUPERVISOR.

Следствие:
- справочник должностей и справочник ролей ведутся отдельно;
- в UI и API они не должны подменять друг друга.

---

## Контракт по CRUD

### 1. Org Units
CRUD допускается для техадмина / привилегированных ролей.

Операции:
- create;
- update;
- deactivate / activate;
- delete — только при допустимости по связям.

### 2. Positions
Полноценный CRUD для привилегированных пользователей.

Операции:
- create;
- read;
- update;
- deactivate / activate;
- delete — только если должность не используется.

### 3. Roles
Полноценный CRUD для привилегированных пользователей.

Операции:
- create;
- read;
- update;
- delete — по правилам ссылочной целостности.

### 4. Personnel
Основной кадровый CRUD.

Операции:
- create;
- read;
- update;
- deactivate;
- transfer;
- delete — только для ошибочно созданных записей и только для привилегированных пользователей.

Важно:
- основной кадровый CRUD выполняется прежде всего из контекста выбранного отделения;
- отдельный общий реестр персонала также допускается.

### 5. Contacts
Полноценный CRUD для контактных данных.

Операции:
- create;
- read;
- update;
- deactivate;
- delete — по правилам ссылочной целостности.

### 6. Working Contacts
Не являются первичным CRUD-справочником.

Операции:
- read;
- фильтрация;
- поиск;
- переход в исходные сущности Personnel / Contacts.

Ввод и редактирование в Working Contacts не являются основным сценарием.

---

## Контракт по правам доступа

### Обычный пользователь
- просмотр отделений в рамках доступного scope;
- просмотр сотрудников выбранного отделения;
- просмотр рабочих контактов в пределах доступного scope;
- без create/update/delete.

### Кадровик / привилегированный пользователь
- просмотр персонала;
- create/update/deactivate personnel;
- перевод сотрудника;
- работа с контактами;
- просмотр и корректировка привязок.

### Техадмин
- полный CRUD по справочникам;
- корректирующие удаления ошибочных записей;
- управление ролями, должностями, отделениями, контактами;
- доступ к производным представлениям и диагностике.

---

## Контракт по UI

# Целевая структура левого меню

Левое меню делится на два уровня:

1. верхний компактный навигационный блок;
2. блоки предметных разделов.

---

## 1. Верхний компактный блок

Оставить:

- Задачи
- Регулярные задачи
- Справочники

Требование:
- сделать этот блок компактнее по высоте;
- уменьшить вертикальные отступы и визуальную высоту карточек;
- поднять нижние предметные блоки выше.

---

## 2. Внутри раздела «Справочники»

Оставить следующие пункты:

- Роли
- Должности
- Контакты
- Персонал
- Группы отделений
- Отделения

Дополнительно:
- Рабочие контакты можно разместить либо здесь же, либо отдельным рабочим разделом ниже.
- Предпочтительный вариант: отдельный пункт `Рабочие контакты` рядом со справочниками, но не смешивать его с первичными справочниками данных.

---

## 3. Блок «Отделения»

Блок отделений остаётся отдельным иерархическим блоком в sidebar.

Назначение:
- навигация по оргструктуре;
- выбор отдела;
- открытие центрального экрана отдела;
- запуск просмотра сотрудников выбранного отделения.

Это основной рабочий вход в список сотрудников по подразделению.

---

# Целевая структура центральных экранов

## 1. Экран «Роли»
Путь:
- `/directory/roles`

Назначение:
- CRUD системных ролей.

Центральный экран:
- список ролей;
- поиск;
- фильтр активных;
- create / edit / delete через drawer.

---

## 2. Экран «Должности»
Путь:
- `/directory/positions`

Назначение:
- CRUD должностей.

Центральный экран:
- список должностей;
- поиск;
- фильтр активных;
- create / edit / delete через drawer.

---

## 3. Экран «Контакты»
Путь:
- `/directory/contacts`

Назначение:
- CRUD контактных данных.

Центральный экран:
- список контактов;
- поиск;
- фильтры;
- create / edit / deactivate;
- переход к связанным кадровым записям при наличии.

---

## 4. Экран «Персонал»
Путь:
- `/directory/personnel`

Назначение:
- общий кадровый реестр.

Центральный экран:
- поиск по ФИО / табельному номеру;
- фильтр по отделению;
- фильтр по должности;
- фильтр по статусу;
- create / edit / deactivate / transfer;
- просмотр карточки сотрудника.

---

## 5. Экран «Рабочие контакты»
Путь:
- `/directory/working-contacts`

Назначение:
- объединённое представление активных сотрудников с контактами.

Центральный экран:
- read-only список;
- поиск;
- фильтры;
- переход в карточку контакта;
- переход в кадровую запись;
- переход в отделение.

Важно:
- основной ввод здесь не выполняется;
- изменения делаются через Contacts и Personnel.

---

## 6. Экран «Группы отделений»
Путь:
- `/directory/department-groups`

Назначение:
- CRUD групп отделений.

---

## 7. Экран «Отделения»
Путь:
- `/directory/org-unit-types` или целевой путь после выравнивания маршрутов

Назначение:
- CRUD справочника отделений / оргединиц;
- работа со структурой.

---

## 8. Экран выбранного отделения
Путь:
- `/directory/org/...` или текущий маршрут дерева отделений

Назначение:
- основной рабочий экран отдела.

Центральный экран должен содержать:

### 8.1. Карточку отделения
- название;
- код;
- группа;
- родительское подразделение;
- статус активности.

### 8.2. Список сотрудников отделения
- ФИО;
- табельный номер;
- должность;
- ставка;
- дата с;
- дата по;
- статус;
- действия.

### 8.3. Действия для привилегированных пользователей
- Добавить сотрудника;
- Изменить;
- Перевести;
- Деактивировать;
- Просмотр карточки;
- Удалить — только для ошибочной записи и только при допустимости.

### 8.4. Действия для обычного пользователя
- только просмотр;
- поиск;
- фильтры;
- открытие карточки сотрудника без прав редактирования.

---

## Принципы навигации

### 1. Personnel и Contacts — отдельные исходные сущности
Их нужно редактировать в отдельных разделах.

### 2. Working Contacts — производный экран
Не использовать как первичный источник ввода.

### 3. Экран отделения — основной контекстный CRUD для сотрудников отдела
Кадровик и привилегированные пользователи работают со списком сотрудников именно из выбранного отдела.

### 4. Roles и Positions не смешивать
Для них должны быть разные экраны и разные API-контракты.

---

## API-направление

Минимально требуется поддержать отдельные маршруты:

### Roles
- `GET /directory/roles`
- `POST /directory/roles`
- `PUT /directory/roles/{id}`
- `DELETE /directory/roles/{id}`

### Positions
- `GET /directory/positions`
- `POST /directory/positions`
- `PUT /directory/positions/{id}`
- `DELETE /directory/positions/{id}`

### Contacts
- `GET /directory/contacts`
- `GET /directory/contacts/{id}`
- `POST /directory/contacts`
- `PUT /directory/contacts/{id}`
- `POST /directory/contacts/{id}/deactivate`
- `DELETE /directory/contacts/{id}`

### Personnel
- `GET /directory/personnel`
- `GET /directory/personnel/{id}`
- `POST /directory/personnel`
- `PUT /directory/personnel/{id}`
- `POST /directory/personnel/{id}/deactivate`
- `POST /directory/personnel/{id}/transfer`
- `DELETE /directory/personnel/{id}`

### Working Contacts
- `GET /directory/working-contacts`
- `GET /directory/working-contacts/{id}` — опционально

### Org Units
- текущие маршруты оргструктуры сохранить;
- дополнить их действиями по кадровому контексту.

---

## Ограничения и правила

1. Физическое удаление personnel и contacts не должно быть основным сценарием.
2. Основной сценарий завершения работы сотрудника — деактивация / дата увольнения.
3. Перевод сотрудника между отделениями должен быть отдельной бизнес-операцией.
4. Рабочие контакты не должны становиться вторым источником истины.
5. Должности не должны использоваться как substitute для ролей.
6. Роли не должны использоваться как substitute для должностей.

---

## Последовательность внедрения

### Этап 1
- сделать компактнее верхний блок sidebar;
- поднять блок отделений выше по экрану.

### Этап 2
- закрепить кадровый CRUD в экране выбранного отделения;
- дать create / edit / deactivate / transfer по сотрудникам отдела.

### Этап 3
- добавить отдельный справочник должностей.

### Этап 4
- добавить отдельный справочник контактов.

### Этап 5
- вывести общий кадровый реестр Personnel.

### Этап 6
- вывести производный экран Working Contacts.

---

## Последствия

### Положительные
- отделяются кадровые, контактные и системные сущности;
- UI становится понятнее;
- снижается риск смешения должностей и ролей;
- экран отдела становится естественной точкой кадровой работы;
- производные рабочие контакты получают понятный источник данных.

### Отрицательные / цена
- потребуется несколько отдельных экранов и CRUD-контрактов;
- потребуется дисциплина по связям между Personnel, Contacts, Positions и Roles;
- понадобится аккуратно провести RBAC для режимов просмотра и редактирования.

---

## Итог

В системе фиксируются четыре базовые первичные сущности:

1. Отделения
2. Персонал
3. Контакты
4. Должности

Дополнительно фиксируются:

5. Роли — как системная сущность прав
6. Рабочие контакты — как производное рабочее представление

Основной кадровый CRUD по сотрудникам отдела выполняется из экрана выбранного отделения.
Общий кадровый реестр Personnel и справочник Contacts ведутся отдельными экранами.
Верхний блок sidebar подлежит уплотнению, чтобы блок отделений находился выше и был визуально ближе к центральному контенту.
