# ADR-024 — Ad hoc / нерегулярные задачи как first-class workflow в общем модуле Tasks

- **Статус:** Proposed
- **Дата:** 2026-03-06
- **Связанные документы:** ADR-020, ADR-021, ADR-022, ADR-023
- **Область:** Backend, UI, RBAC, Tasks FSM, Events/Notifications

## Контекст

В системе уже поддержан устойчивый контур для регулярных задач:

- шаблон хранится в `regular_tasks`;
- экземпляры задач создаются генератором;
- задача проходит через общий модуль `Tasks`;
- жизненный цикл задачи поддерживается через FSM, события, аудит и доставки уведомлений.

В ходе тестирования выявлен отдельный прикладной сценарий: **нерегулярные / разовые / ad hoc задачи**, которые создаются вручную:

- директором;
- заместителем;
- руководителем подразделения;
- администратором поддержки по поручению руководителя.

Такие задачи в реальной эксплуатации неизбежны и не должны рассматриваться как исключение или технический обходной путь.

Одновременно установлено, что ручное создание задачи через минимальный `INSERT` без явного постановщика/согласующего приводит к неустойчивому поведению маршрутизации, в частности к ошибкам вида:

> Cannot resolve approver: `regular_tasks.target_role_id` is `NULL`, org chain HEAD not found, and task initiator fallback is empty.

Следовательно, для ad hoc задач требуется явный архитектурный контракт.

## Проблема

Нельзя проектировать разовые задачи как побочный случай `regular_task_id IS NULL`.

Если ad hoc задача не имеет явной модели происхождения и согласования, система вынуждена полагаться на fallback-логику:

1. попытка разрешить согласующего из `regular_tasks`;
2. попытка вывести согласующего из оргцепочки;
3. fallback на `initiator_user_id`.

Такой подход:

- делает поведение хрупким;
- затрудняет UX и диагностику;
- усложняет RBAC;
- приводит к 500-ошибкам вместо управляемой бизнес-валидации;
- смешивает происхождение задачи с её жизненным циклом.

## Решение

### 1. Единый модуль Tasks для всех экземпляров задач

**Регулярные и нерегулярные задачи должны жить в одном модуле `Tasks`.**

Это означает:

- одна доменная сущность `Task`;
- единый список задач;
- единая карточка задачи;
- единый аудит;
- единые события и доставки уведомлений;
- единая FSM;
- единый архив.

Разделение проходит **не по модулю**, а **по способу происхождения задачи**.

### 2. Разделение шаблонов и экземпляров

Фиксируется следующая доменная модель:

- `regular_tasks` — шаблоны и правила генерации;
- `reporting_periods` — периоды для плановых задач;
- `tasks` — все экземпляры задач (regular и ad hoc);
- `task_reports` — отчёты по задачам;
- `task_events` / `task_audit` / `task_event_deliveries` — события, аудит, доставки.

### 3. Ad hoc задача — first-class workflow

Ad hoc задача — это самостоятельный поддерживаемый сценарий, а не fallback-режим regular task.

Для ad hoc задачи:

- `regular_task_id` может быть `NULL`;
- маршрут согласования должен быть задан явно или выводиться по ясному правилу;
- отсутствие конфигурации маршрута должно приводить к управляемой ошибке 4xx, а не к 500.

## Архитектурный контракт

### 1. Общие свойства задачи

Любая задача в системе, независимо от происхождения, имеет:

- `title`;
- `description`;
- `status_id`;
- `executor_role_id` и/или иную исполнительскую адресацию по принятой модели;
- `assignment_scope`;
- `created_at` / `updated_at`;
- аудит;
- события;
- уведомления;
- опционально отчёт.

### 2. Различие по происхождению

Фиксируются минимум два типа происхождения:

- `regular` — задача создана генератором из `regular_tasks`;
- `adhoc` — задача создана вручную из UI/API.

Рекомендуемые поля в `tasks`:

- `task_kind` — `regular | adhoc`;
- `source_kind` — `regular_task | manual | bot | import`;
- `created_by_user_id` — кто технически создал запись;
- `initiator_user_id` — кто является постановщиком по бизнес-смыслу;
- `approver_user_id` — кто принимает/отклоняет отчёт;
- `requires_report` — нужен ли отчёт;
- `requires_approval` — нужен ли этап согласования.

Минимальный обязательный набор для первого этапа внедрения:

- `created_by_user_id`;
- `approver_user_id`;
- `task_kind`;
- `requires_report`;
- `requires_approval`.

## Правила для ad hoc задач

### 1. Правила создания

При создании ad hoc задачи должны быть явно определены:

- постановщик (`initiator_user_id`);
- исполнитель (`executor_role_id` или иной поддержанный тип адресации);
- срок, если он нужен бизнес-процессом;
- требуется ли отчёт;
- требуется ли согласование;
- кто принимает результат (`approver_user_id`), если согласование требуется.

### 2. Обязательные инварианты

Для задачи `task_kind = 'adhoc'`:

1. `title` обязателен;
2. исполнитель обязателен;
3. `initiator_user_id` обязателен;
4. если `requires_approval = true`, то должен быть задан `approver_user_id` **или** должен быть явно разрешён fallback на `initiator_user_id`;
5. если `requires_report = false`, задача может завершаться без отчёта;
6. отсутствие согласующего не должно приводить к 500.

### 3. Политика резолюции согласующего

При действии `report` согласующий определяется в следующем порядке:

1. `tasks.approver_user_id`, если заполнен;
2. `tasks.initiator_user_id`, если политика допускает fallback на постановщика;
3. конфигурация из `regular_tasks`, если задача regular и там явно задан маршрут;
4. оргцепочка, только если это предусмотрено и может быть выполнено детерминированно;
5. иначе вернуть понятную бизнес-ошибку 4xx: **Approver is not configured for this task**.

Для `adhoc` задачи приоритет должен быть отдан **явным полям самой задачи**, а не выводу из шаблона.

## FSM-контракт

Фиксируются следующие допустимые контуры:

### Сценарий A. Задача с отчётом и согласованием

```text
IN_PROGRESS
  -> REPORT_SUBMITTED
WAITING_APPROVAL
  -> APPROVE -> DONE
  -> REJECT  -> IN_PROGRESS
```

### Сценарий B. Задача без согласования

```text
IN_PROGRESS
  -> COMPLETE -> DONE
```

### Сценарий C. Задача с отчётом, но без approval-этапа

```text
IN_PROGRESS
  -> REPORT_SUBMITTED -> DONE
```

FSM должна учитывать не только статус, но и конфигурацию задачи:

- `task_kind`;
- `requires_report`;
- `requires_approval`.

## Контракт API

### 1. Общий принцип

Все экземпляры задач отображаются и обрабатываются в общем модуле `Tasks`.

### 2. Создание ad hoc задачи

Рекомендуется ввести отдельный сценарий создания, например:

```text
POST /tasks/manual
```

Пример тела запроса:

```json
{
  "title": "Подготовить разовый свод по административному блоку",
  "description": "По поручению директора",
  "executor_role_id": 10,
  "initiator_user_id": 14,
  "approver_user_id": 14,
  "assignment_scope": "role",
  "due_date": "2026-03-10T18:00:00+05:00",
  "task_kind": "adhoc",
  "source_kind": "manual",
  "requires_report": true,
  "requires_approval": true
}
```

Пример для ad hoc задачи без согласования:

```json
{
  "title": "Уточнить список оборудования",
  "description": "Разовое поручение",
  "executor_role_id": 10,
  "initiator_user_id": 14,
  "assignment_scope": "role",
  "task_kind": "adhoc",
  "source_kind": "manual",
  "requires_report": false,
  "requires_approval": false
}
```

### 3. Валидация API

При создании manual/ad hoc задачи API обязано:

- проверять наличие постановщика;
- проверять наличие исполнителя;
- при `requires_approval = true` требовать `approver_user_id` либо фиксированный допустимый fallback на постановщика;
- отклонять неконсистентную конфигурацию как 4xx-ошибку с понятным текстом.

## Контракт UI

Разовые задачи не выносятся в отдельный модуль.

Они отображаются в общем разделе **Все задачи / Tasks** с фильтрами по происхождению и состоянию.

Рекомендуемые UI-фильтры:

- Все;
- В работе;
- На согласовании;
- Отработано;
- Отклонено / Возвращено на доработку;
- Регулярные;
- Разовые.

В форме создания разовой задачи UI должен явно показывать:

- Постановщик;
- Исполнитель;
- Кто принимает результат;
- Срок;
- Нужен ли отчёт;
- Нужно ли согласование.

Если задачу создаёт администратор поддержки по поручению директора, допустима схема:

- `created_by_user_id = support/admin`;
- `initiator_user_id = director`;
- `approver_user_id = director`.

## Последствия

### Положительные

- регулярные и нерегулярные задачи образуют единый пользовательский контур;
- снимается зависимость ad hoc задач от `regular_task_id`;
- маршрут согласования становится явным и предсказуемым;
- уменьшается количество скрытых fallback-веток;
- система становится ближе к реальному управленческому процессу.

### Отрицательные / стоимость изменений

- потребуется расширение схемы `tasks`;
- потребуется корректировка `tasks_fsm.py`;
- потребуется новый API-сценарий создания manual task;
- потребуется доработка UI-формы создания задачи;
- потребуется обновление прав и валидации RBAC для ручной постановки задач.

## Этапы внедрения

### Этап 1. Минимально рабочий

1. Добавить в `tasks`:
   - `created_by_user_id`;
   - `approver_user_id`;
   - `task_kind`;
   - `requires_report`;
   - `requires_approval`.
2. Изменить FSM так, чтобы при `report` сначала использовался `approver_user_id`.
3. Запретить 500 при отсутствии approver; возвращать 4xx с понятной бизнес-ошибкой.

### Этап 2. Полноценный контур

1. Добавить API `POST /tasks/manual`.
2. Добавить UI-форму создания разовых задач.
3. Добавить фильтры `regular / adhoc` в модуле задач.
4. Доработать UX для состояний «отклонено» и «возвращено на доработку».

## Принятое решение

Принять ad hoc / нерегулярные задачи как **first-class workflow** в общем модуле `Tasks`.

- Экземпляры regular и ad hoc задач хранятся в едином контуре `tasks`.
- Модуль `regular_tasks` остаётся модулем шаблонов и генерации.
- Для ad hoc задач маршрут должен задаваться явно и не должен зависеть от неустойчивых fallback-веток.
- Отсутствие согласующего для ad hoc задачи должно считаться ошибкой конфигурации задачи, а не исключением инфраструктуры.

## Открытые вопросы

1. Нужен ли `approver_role_id` помимо `approver_user_id`?
2. Нужна ли поддержка прямого назначения задачи на пользователя, а не только на роль?
3. Нужен ли отдельный статус/признак «возвращено на доработку» для UX, если текущий статус снова `IN_PROGRESS`?
4. Нужно ли делать `period_id` обязательным для всех ad hoc задач или допускать служебный/null-режим?
5. Нужно ли отделять `created_by_user_id` и `initiator_user_id` на уровне UI-роли и аудита как разные сущности во всех сценариях?
