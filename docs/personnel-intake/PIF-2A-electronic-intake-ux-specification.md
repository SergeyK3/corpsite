# PIF-2A — Electronic Personnel Intake UX Specification

## Status

**Active (UX target + production divergence documented)** — UX specification initiated 2026-07-08; updated 2026-07-24.

Этот документ описывает **целевой candidate-side UX** для согласования с HR и статический прототип [demo/](./demo/). **Production** (`corpsite-ui`) реализует **упрощённый 9-шаговый wizard**, не полный сценарий из §4.

## 1. Purpose

### 1.1. Зачем этот документ

Зафиксировать **пользовательский опыт кандидата** при заполнении электронной анкеты для оформления на работу — от первого открытия защищённой ссылки до успешной отправки сведений в отдел кадров.

Документ предназначен для:

- демонстрации HR практического концепта продукта;
- согласования состава полей и обязательности;
- передачи в дизайн и разработку (PIF-4) без двусмысленностей.

### 1.2. Что это за продукт

**Современная пошаговая анкета**, которая заменяет бумажное заполнение и ручной перенос данных HR в Excel и кадровую систему. Кандидат сам вводит сведения; HR позже только проверяет полноту и корректность (будущие WP — не в scope этого документа).

### 1.3. Границы scope

Спроектирован **target candidate-side UX** для HR-согласования и прототипа [demo/](./demo/). **Production** реализует candidate + HR on-behalf в `corpsite-ui` (см. §1.4). HR-side review детально не специфицирован — упоминается в demo script.

### 1.4. Production vs this specification

| Aspect | Target UX (this document) | Production (`corpsite-ui`, 2026-07-24) |
|--------|---------------------------|----------------------------------------|
| Step count | 10 тематических шагов + Welcome + Success (§4 overview) | **9 рабочих шагов** + «Проверка» (`INTAKE_STEPS`) |
| Welcome / Success | Отдельные экраны | Нет — сразу форма |
| Identity document | Отдельный шаг 3 | Объединено с персональными/контактными полями |
| Military | Шаг 8 (условный) | ✅ Отдельный шаг «Воинский учёт» |
| Photo | **3×4 см** (target UX + production PDF frame) | ✅ Upload; рамка **3×4 см** в preview-PDF |
| PDF | Не описан (future) | ✅ **Preview-PDF** на шаге «Проверка» (pre-commit) |
| Form engine | Conceptual cards / wizard | Static React components |
| Declarations checkboxes | На экране проверки | Review summary + submit (partial) |

**Как читать документ:** screen-by-screen спецификация §4 = **target** для HR workshop и [demo/](./demo/); production — **subset / merge** шагов, уже частично реализованный в коде.

## 2. User scenario

### 2.1. Контекст

HR готовит приём нового сотрудника. Вместо выдачи бумажной анкеты HR отправляет кандидату **защищённую ссылку** (WhatsApp, Telegram, email — канал доставки вне scope). Кандидат открывает ссылку на телефоне или компьютере и заполняет анкету самостоятельно.

### 2.2. Happy path

```text
HR создаёт приглашение и отправляет ссылку
  ↓
Кандидат открывает ссылку
  ↓
Welcome → «Начать заполнение»
  ↓
Шаги 1–9: по одной теме на экран (основные сведения → … → дополнительные)
  ↓
Шаг 10: «Проверка заполнения»
  ↓
«Отправить в отдел кадров»
  ↓
«Анкета отправлена в отдел кадров»
```

### 2.3. Прерывание и возобновление

Кандидат может закрыть анкету на любом шаге. При повторном открытии той же ссылки система **концептуально** возвращает его на последний незавершённый шаг с сохранёнными данными (механизм хранения — out of scope).

### 2.4. Акторы

| Актор | Роль в этом документе |
|-------|----------------------|
| **Кандидат / новый сотрудник** | Единственный пользователь спецификации |
| **HR** | Получатель отправленной анкеты; проверяет offline (demo script) |

## 3. UX principles

| # | Principle | Реализация в анкете |
|---|-----------|---------------------|
| 1 | Мастер по шагам | 10 тематических шагов + проверка; не одна длинная форма |
| 2 | Один экран = одна тема | Каждый шаг — один смысловой блок |
| 3 | Карточки, не таблицы | Образование, работа, языки — карточки с «Добавить ещё» |
| 4 | Явное добавление записей | «Добавить образование», «Добавить место работы» |
| 5 | Видимый прогресс | «Шаг X из 10», процент заполнения, список пробелов |
| 6 | Автосохранение | Концептуально при каждом «Далее» и при изменении поля |
| 7 | Защищённая ссылка | Анкета открывается только по персональной ссылке |
| 8 | RU / KZ | Документ на русском; интерфейс готов к локализации (будущее) |
| 9 | Простой язык | Без технических терминов; понятно без IT-подготовки |
| 10 | Self-service | Кандидат заполняет всё сам; HR только проверяет |

### 3.1. Общие элементы каждого шага (кроме Welcome и Success)

| Element | Description |
|---------|-------------|
| **Заголовок шага** | Название темы (например, «Основные сведения») |
| **Индикатор прогресса** | «Шаг 2 из 10» + полоска прогресса |
| **Процент заполнения** | Обновляется по мере заполнения обязательных полей всей анкеты |
| **Тело формы** | Поля текущего шага |
| **Навигация** | «Назад» (кроме шага 1), «Далее» |
| **Подсказка внизу** | «Данные сохраняются автоматически. Вы можете продолжить позже.» |

### 3.2. Обработка ошибок на шаге

При нажатии «Далее» с незаполненными обязательными полями:

- поля подсвечиваются;
- под полем — короткое сообщение («Укажите фамилию»);
- прокрутка к первому проблемному полю;
- переход на следующий шаг **не выполняется**.

## 4. Screen-by-screen specification

### Экран 0. Welcome — старт анкеты

| Attribute | Value |
|-----------|-------|
| **Название** | Анкета нового сотрудника |
| **ID** | `welcome` |
| **Назначение** | Приветствие; мотивация начать; снятие тревоги по времени и сохранению |

#### Что видит пользователь

- Заголовок: **Анкета нового сотрудника**
- Подзаголовок: **Анкета для оформления на работу**
- Короткий текст (3–4 строки):

  > Здравствуйте! Заполните анкету, чтобы мы могли оформить ваш приём на работу.
  > Примерное время: **20–30 минут**.
  > Вы можете сохранить прогресс и **продолжить позже** по этой же ссылке.
  > После отправки сведения будут направлены **в отдел кадров**.

- Иконка или нейтральная иллюстрация (опционально, без перегруза)
- Кнопка: **Начать заполнение**

#### Поля

Нет полей ввода.

#### Действия

| Действие | Результат |
|----------|-----------|
| «Начать заполнение» | Переход на экран 1 «Основные сведения» |

#### Зачем HR

Кандидат понимает цель и объём работы; снижается число бросивших анкету на полпути.

## 6. Validation rules

### 6.1. Общие правила

| Rule ID | Правило | Сообщение пользователю |
|---------|---------|------------------------|
| V-01 | Обязательные поля не пусты | «Заполните это поле» |
| V-02 | ИИН — ровно 12 цифр | «ИИН должен содержать 12 цифр» |
| V-03 | Email — корректный формат (если заполнен) | «Проверьте адрес email» |
| V-04 | Телефон — не пустой где обязателен; мин. 10 цифр | «Укажите номер телефона» |
| V-05 | Дата — корректный формат ДД.ММ.ГГГГ | «Укажите дату в формате ДД.ММ.ГГГГ» |
| V-06 | Дата не в будущем (где применимо) | «Дата не может быть в будущем» |
| V-07 | Дата окончания ≥ дата начала | «Дата окончания не может быть раньше даты начала» |
| V-08 | Год окончания образования ≤ текущий год | «Укажите корректный год окончания» |
| V-09 | «Работаю по настоящее время» → дата окончания необязательна | — |
| V-10 | Адрес совпадает → копирование адреса регистрации | — |
| V-11 | Необязательный раздел → явный «Нет сведений» | «Выберите «Нет сведений» или добавьте запись» |
| V-12 | Загрузка файла — допустимый формат и размер | «Файл слишком большой или неверный формат» |
| V-13 | Срок действия документа ≥ дата выдачи | «Проверьте срок действия документа» |
| V-14 | Минимум 1 запись образования | «Добавьте сведения об образовании» |
| V-15 | Трудовая: опыт или «Нет опыта» | «Укажите место работы или выберите «Нет опыта»» |
